# MediaMTX Connect — Feature Ideas Catalog

> **Status: ideas, not implemented.** Brainstorm catalog generated from MediaMTX's config schema, OpenAPI spec, README, and source. **Nothing here is shipped.** The actual feature inventory lives in [`docs/FEATURES.md`](../FEATURES.md). Use this as a menu, not a roadmap — entries are unprioritized and may never be built.

A long-list of feature ideas for the **MediaMTX Connect** web UI, generated by reading every section of MediaMTX's `mediamtx.yml` default config, the v3 OpenAPI spec, the README, and the existing `docs/FEATURES.md` to avoid duplicating shipped work.

## How this was built

To stay exhaustive without drowning in one giant document, the surface area was split into five domains. One research pass per domain, each grounded in the upstream MediaMTX source (config schema, OpenAPI, internal Go packages where relevant). Every idea cites the MediaMTX building block it relies on (config key, API endpoint, hook name) so it's traceable to something real.

## How to use this catalog

1. **Pick a domain** the team wants to invest in next.
2. **Read that file end-to-end** — it groups ideas so adjacent bullets often share a UI surface and can ship together.
3. **Triage each section** with the simple rubric below. A single bullet rarely justifies a feature; a section often does.

### Triage rubric

| Axis | Quick test |
|------|-----------|
| **Reach** | Does every MediaMTX user benefit, or only a niche (e.g. SRT-only, Pi-only)? |
| **Existing API support** | Does MediaMTX already expose the data/endpoints, or would we need to ask upstream? Existing-only is much cheaper. |
| **Net-new infra** | Does it require new long-running components (Prometheus, queue, message bus)? Prefer ideas that fit our existing Hono + JSON-config-store + node-cron shape. |
| **Maintenance contract** | Will the feature need to track MediaMTX schema changes? Pin to stable APIs first; defer features built on `deprecated: true` fields. |
| **Auth gating** | Does it need authn/authz before it's safe to ship? If so, it's blocked on `04-auth-security-hooks.md` work. |

## Cross-cutting ideas

These don't fit cleanly inside any one domain — they're product-shaped or infrastructure-shaped, touching multiple files at once. They're often the highest-leverage additions because they unlock a category, not just a feature.

### Onboarding & first-run experience

- **Setup wizard** — replace today's three-field config form with a multi-step wizard: detect the running MediaMTX, test API reachability, prompt for the remote URL, walk the user through publishing their first stream. Wraps `v3/configGlobalGet` + `v3/pathsList` + a "publish a test stream" page.
- **Doctor / preflight page** — one screen that runs every diagnostic in the app (API reach, recordings dir writable, screenshots dir writable, ffmpeg present, MediaMTX version supported, time-sync, port binds OK) with green/red rows. Mirrors `pnpm doctor`-style ergonomics.
- **Sample stream provisioner** — one click installs the `examples/fake-streams/` rig so a new user has something to look at immediately.
- **Empty-state CTAs everywhere** — every empty list (no streams, no recordings, no users, no hooks) ships with a "do this next" CTA that deep-links into the right wizard.

### Multi-tenancy & sharing

- **Public read-only gallery** — opt-in per path, exposes a tokenless `/watch/<slug>` page with HLS playback, no auth. Effectively turns MediaMTX Connect into a small streaming site builder.
- **Per-path share links with expiry** — short-lived signed URL that hides the underlying MediaMTX URL, useful for sending a stream to one external viewer without changing config.
- **Embed builder** — generates iframe + `<video>` snippets for any path with controls (autoplay, mute, theme, branded poster).
- **Multi-organization mode** — separate logical "tenants" inside one Connect instance; each tenant maps to a subset of paths and users. Backed by a new `Tenant` table + per-tenant ACLs (`authInternalUsers`).
- **White-label theming** — runtime-overridable brand color, logo, app name; each tenant can have its own.

### Mobile & remote viewing

- **Native PWA push notifications** — on path-down, recording-failure, security alerts. Uses the existing `service-worker.tsx` + Web Push.
- **Mobile-first stream viewer** — gestures (swipe to switch streams, pinch to zoom, double-tap fullscreen), PiP, "bring to front" to OS controls.
- **Offline browse** — service worker caches the recording index + thumbnails so on-the-train browsing works.
- **Background recording sync** — flag a recording for offline keep; the SW pre-downloads it.

### AI / ML overlays

These fit MediaMTX's pluggable model (data goes out via a hook; events come back via the API or a webhook).

- **Motion detection on segments** — every `runOnRecordSegmentComplete` triggers a server job that runs ffmpeg motion detection and writes events to a new `Event` table; UI overlays them on the timeline.
- **Object detection (YOLO / DETR) integration** — same pattern, but with an external worker (e.g. a CompreFace / Frigate / Coral TPU sidecar). The UI just consumes the results.
- **Speech transcription per recording** — Whisper (local or API) generates SRT files alongside MP4s; UI shows captions and lets the user search transcripts globally.
- **Smart thumbnails** — pick the most "interesting" frame (highest motion / most detected objects) instead of frame-zero.
- **NSFW / safety filter** — block the live thumbnail/playback for paths flagged unsafe by a model, with admin override.
- **Auto-tagging** — apply tags ("person", "car", "package") from object detection; let users filter recordings by tag.
- **Scene change detection** — break one recording into auto-chapters at scene cuts.
- **Anomaly alerts** — train a tiny model on bytes-in/sessions per hour-of-week; alert when actual diverges. Plays nicely with `05-observability-and-ops.md`'s metrics work.

### Editing, transcoding, ffmpeg-as-a-service

- **Server-side clip exporter** — UI picks in/out points; server runs ffmpeg with `-c copy` when possible, otherwise a re-encode job; user gets an MP4. Builds on `playback /get`.
- **Mosaic / multiview generator** — combine N live streams into a single `2x2` or `3x3` mosaic stream that MediaMTX republishes. Use ffmpeg with `xstack` + a managed second path.
- **Transcoding workers** — make a sidecar service (a "studio") that subscribes to a path, transcodes, republishes to a `path-low` and `path-high` for ABR-style readers.
- **GIF / animated preview generator** — auto-generates a 5-second looping animated preview per recording for the index cards.
- **Burn-in overlays** — timestamps, watermark, path name overlaid on exported clips.
- **Concatenate recordings** — multi-select on the recordings index, server stitches into one MP4.

### Browser publishing studio (deeper than WHIP alone)

- **Studio mode** — webcam + screen share + RTMP/SRT pull, mixed in the browser via WebRTC + canvas, then published to MediaMTX via WHIP. Roughly OBS-in-the-browser, scoped to a single output.
- **Push-to-talk audio** — open a one-way audio channel back to a Pi/IP camera that listens via RTSP backchannel.
- **Drawing / annotation overlay** — draw on the live view; the strokes are republished as a separate path or burned into a transcoded copy.

### Integrations marketplace

- **Home Assistant integration** — expose MediaMTX paths as HA cameras, surface HA state in the Connect UI (lights, motion sensors next to a stream).
- **Frigate / Shinobi / ZoneMinder bridge** — import their motion events; treat MediaMTX Connect as the unified viewer.
- **Discord / Slack / Teams / Telegram** — drop-in alert templates (covered in `04`), but also full bot interfaces ("`/clip stream-foo last 30s`").
- **OBS WebSocket bridge** — show OBS scenes/sources next to the matching MediaMTX path.
- **Dashboard tile / iframe widget** — a tiny endpoint that renders a single stream in an embeddable iframe sized for Grafana or a kiosk.
- **Webhook IFTTT-style rules engine** — "if path X goes offline for >2 min, call this webhook with this payload."

### Theming, accessibility, i18n

- **Per-user theme + density** — extend the existing `ModeToggle` and `liveDensity` localStorage keys into a full Settings page for accent color, density, motion-reduction, font size.
- **Captions/subtitles** in the player — wire transcripts (above) into the HLS.js player.
- **Audio description track** — surface secondary audio if MediaMTX exposes it.
- **High-contrast and reduce-motion modes** — explicit toggles that override the in-app ThemeProvider for accessibility users.
- **Locale expansion** — beyond `en`/`es`, ship `fr`/`de`/`ja`/`zh`/`pt` from the same `messages/` system. Add a community-translation contribution flow (e.g. Crowdin).
- **RTL support** — Arabic/Hebrew layouts driven by the client-side locale setting.
- **Keyboard shortcuts** — `?` opens a cheatsheet; `j/k` between streams, `space` play/pause, `g r` go to recordings.
- **Screen-reader optimized live view** — narrate stream-up/stream-down state changes via ARIA live regions.

### Plugin / extension system

- **In-app plugin SDK** — declare a small `Plugin` interface (`registerSidebarItem`, `registerStreamCardAction`, `registerHookHandler`) that lets third parties extend the UI without forking. Loaded from a `plugins/` directory.
- **Hook plugin marketplace** — curated `runOn*` scripts that users can install with one click (covered narrower in `04`).
- **Slash-command palette** — a `cmdk`-style command bar (the project already imports shadcn `Command`) that any plugin can register actions into.

### Data model & developer experience

- **Postgres / MySQL option** — introduce a database layer so larger deployments can grow beyond the JSON config store without forking. Docs + Marketplace integration with Vercel Neon Postgres.
- **Full GraphQL or tRPC layer** — for plugin authors and external automation; sits in front of the existing oRPC procedures.
- **OpenAPI for the Connect API** — first-class spec for `/api/health`, `/api/.../first-screenshot`, etc., so external tooling can integrate.
- **CLI** — `mediamtx-connect` CLI that wraps the existing oRPC procedures (config get/set, recording list/delete, alert rules) for ops automation.
- **Terraform / Pulumi provider** — declare paths, users, and alerts as infrastructure-as-code.

### Competitive / category-shaped ideas

These reframe Connect against existing categories rather than additive features.

- **NVR mode** — package the app + MediaMTX + a default config tuned for IP-camera DVR (24/7 recording, motion-tagged events, calendar UI). Compete with Frigate / Shinobi.
- **Streaming-platform mode** — public gallery, per-stream chat, viewer count, embed builder. Compete with Owncast.
- **Surveillance kiosk mode** — fullscreen multi-cam, no chrome, auto-cycle. Compete with low-end CCTV walls.
- **Demo / showcase mode** — auto-generate a public link with a curated subset of streams + recordings for sharing with stakeholders.

### Documentation & community

- **In-app help drawer** — context-sensitive snippets ("what is `recordPath`?") pulled from MediaMTX docs.
- **"Why is this offline?" troubleshooter** — reads recent log lines for the path, matches them against a corpus of known-error patterns, suggests a fix.
- **Config reference inline** — every form field links to the MediaMTX docs for that key.
- **Changelog viewer for upstream MediaMTX** — pulls release notes when the version changes; flags breaking changes that affect the user's config.

### Hardware-specific

- **Raspberry Pi camera (covered deeply in `01`)** — the rpiCamera section alone is 13 ideas because every camera knob is exposed via path config.
- **NVIDIA / Intel QuickSync transcoding** — UI for picking hardware acceleration for any transcode job (mosaic, exporter, transcoder).
- **Coral / Hailo TPU detection sidecar** — official sidecar image bundled with Connect for object detection on the same box.
- **USB capture card support** — guided wizard for ingesting from a UVC device via ffmpeg → RTSP.
